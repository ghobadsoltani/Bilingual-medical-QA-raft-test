# Retrieval Optimization Guide\n\n## Core Principle\n\nGood retrieval = Good QA. The best language model can't compensate for poor document retrieval. This guide covers practical strategies for improving medical document retrieval with limited resources.\n\n## 1. Pipeline Architecture\n\n### Simple Pipeline (Good for Prototypes)\n\n```\nQuery → Embedding → Vector Search (FAISS) → Top-5 docs → LLM → Answer\n```\n\n**Pros:** Simple, fast  \n**Cons:** Limited by embedding quality\n\n### Hybrid Pipeline (Recommended for Production)\n\n```\nQuery → [Embedding Path] → Top-20\n         [BM25 Path]      → Merge\n                              ↓\n                         Top-10 (reranked)\n                              ↓\n                          LLM Answer\n```\n\n**Pros:** Best quality, captures both semantic + keyword matches  \n**Cons:** 2x retrieval latency (mitigate with parallel execution)\n\n### Advanced Pipeline (If Quality Still Limited)\n\n```\nQuery → Hybrid Retrieval (50 docs) → Cross-Encoder Reranker → Top-5 → LLM\n```\n\n**Pros:** Near-SOTA retrieval quality  \n**Cons:** Significant latency cost (cross-encoder is slow)\n\n## 2. Embedding Selection for Medical Text\n\n### Benchmark: Medical Retrieval Tasks\n\nOn clinical query datasets:\n- **all-MiniLM-L6-v2:** 65% precision@10 (baseline)\n- **all-mpnet-base-v2:** 72% precision@10 (+7%)\n- **PubMedBERT-embeddings:** 78% precision@10 (+13%)\n\n### Selection Decision Tree\n\n```\nDo you have GPU + time budget?\n├─ Yes → Use PubMedBERT or domain-specific model\n├─ No → Use all-MiniLM-L6-v2\n\nIs latency critical? (< 100ms)\n├─ Yes → all-MiniLM-L6-v2\n├─ No → all-mpnet-base-v2\n\nIs document corpus domain-specific?\n├─ Clinical notes → Domain-specific embedding\n├─ General medical → all-mpnet-base-v2\n```\n\n## 3. Chunking Strategy\n\n### Problem\nHow you split documents directly impacts retrieval quality.\n\n### Strategies by Document Type\n\n#### Clinical Notes (Unstructured)\n**Best: Sentence-based with minimum 80 tokens, maximum 200 tokens**\n```python\nChunk: \"Patient presents with chest pain radiating to left arm. \nBM vitals stable. ECG shows ST elevation in V1-V3.\"\n(~30 tokens = too small)\n\nChunk: \"Patient presents with chest pain radiating to left arm.\nBM vitals stable. ECG shows ST elevation in V1-V3.\nTroponin elevated at 0.8 ng/mL. Clinical impression: STEMI.\"\n(~50 tokens = good)\n```\n\n#### Research Abstracts (Structured)\n**Best: Fixed-size 256-512 tokens with 0% overlap**\n- Structured format (intro-methods-results) benefits from larger chunks\n- Avoid overlap (no redundancy)\n\n#### Medical Guidelines (Hierarchical)\n**Best: Respect document structure**\n```\nSection: \"Sepsis Management\"\n  Subsection: \"Antibiotics\"\n    Content: \"First-line antibiotics...\"\n    \nChunk by: Subsection (may be 100-500 tokens depending)\n```\n\n### Chunking Best Practices\n\n1. **Include context:** Each chunk should be self-contained\n   ```\n   ❌ \"given Q6H\" (what medication?)\n   ✅ \"Vancomycin 1g IV Q6H for MRSA pneumonia\"\n   ```\n\n2. **Preserve sentence boundaries:** Split at periods, not mid-sentence\n\n3. **Test empirically:** Different corpora need different chunk sizes\n   - Clinical: 100-150 tokens\n   - Research: 256-512 tokens\n   - Guidelines: 200-400 tokens\n\n## 4. BM25 (Keyword Search)\n\n### When BM25 Excels\n- Exact terminology matches: \"Vancomycin\" → finds all vancomycin mentions\n- Acronyms: \"STEMI\" → specific cardiac condition\n- Medication names, dosages, units\n\n### When BM25 Fails\n- Synonyms: \"Heart attack\" vs \"Myocardial infarction\"\n- Paraphrasing: \"Reduced kidney function\" vs \"Renal impairment\"\n- Implicit meaning: \"Elevated BP\" vs \"Hypertension\"\n\n### Hybrid Ranking Strategy\n\n**Reciprocal Rank Fusion (RRF):**\n```python\ndef rrf_score(bm25_rank, semantic_rank, k=60):\n    return 1/(k + bm25_rank) + 1/(k + semantic_rank)\n```\n\n**Advantages:**\n- No need to normalize scores (handles different scales)\n- Simple, effective, requires no training\n\n**Weighted Combination Alternative:**\n```python\nscore = 0.4 * normalize(bm25_score) + 0.6 * cosine_similarity\n```\n(Requires tuning weights per domain)\n\n## 5. Retrieval Evaluation\n\n### Without Ground Truth Labels\n\n**Proxy metrics:**\n1. **Cosine similarity distribution** – Are relevant docs separated from noise?\n2. **Query-document overlap** – Does retrieved doc contain query terms?\n3. **Manual spot-check** – Sample 20 queries, review top-5 results\n\n### With Labeled Data (Gold Standard)\n\n**Metrics:**\n- **Precision@K:** % of top-K results that are relevant\n  - Precision@10 = (# relevant in top-10) / 10\n- **Recall:** % of all relevant docs that are retrieved\n- **NDCG:** Discounted gain (early results matter more)\n- **Mean Reciprocal Rank (MRR):** Rank position of first relevant doc\n\n### Practical Evaluation Workflow\n\n```\n1. Take 20 representative queries from your domain\n2. For each, manually identify relevant documents\n3. Run retrieval pipeline, measure Precision@10\n4. If < 70%: investigate failure modes\n   - Missing embeddings? Add them.\n   - Chunking too small? Increase size.\n   - Need hybrid? Add BM25.\n5. Iterate until Precision@10 ≥ 80%\n```\n\n## 6. Optimization Checklist\n\n- [ ] **Embeddings:** Using domain-appropriate model?\n- [ ] **Chunking:** Tested multiple strategies on YOUR data?\n- [ ] **Indexing:** Documents encoded and indexed? (FAISS, Weaviate, Pinecone)\n- [ ] **Hybrid:** Running both semantic + BM25?\n- [ ] **Evaluation:** Tested on realistic queries? Precision@10 ≥ 80%?\n- [ ] **Caching:** Are embeddings cached, not recomputed per query?\n- [ ] **Latency:** Full pipeline < 500ms for production?\n\n## 7. Common Mistakes\n\n| Mistake | Impact | Fix |\n|---------|--------|-----|\n| Too-large chunks | Diluted relevance signals | Use 100-200 token chunks |\n| Semantic-only | Misses exact terminology | Add BM25 ranking |\n| No evaluation | Flying blind | Manually test 20 queries |\n| Recomputing embeddings | 10x latency overhead | Cache embeddings |\n| Ignoring domain | Generic embeddings fail | Use domain-specific model |\n| No preprocessing | Acronyms, abbreviations unsupported | Expand abbreviations |\n\n## 8. Medical-Specific Considerations\n\n### Clinical Terminology\n- Lots of acronyms (STEMI, ARDS, DVT)\n- Solution: Expand acronyms in preprocessing\n  ```\n  \"STEMI\" → \"STEMI (ST-elevation myocardial infarction)\"\n  ```\n\n### Dosage Information\n- Important but sparse in documents\n- Solution: Extract structured (drug, dose, frequency, indication)\n- Can't rely on embedding similarity alone for dosage queries\n\n### Negations\n- \"No signs of infection\" vs \"Signs of infection\"\n- Embeddings sometimes ignore negation\n- Solution: Use negation-aware preprocessing or models trained on clinical text\n\n## References\n\n- **Hybrid Search:** \"Hybrid Search at Scale\" (Vespa blog)\n- **Chunking:** Domain-dependent, test empirically\n- **RRF:** \"Reciprocal Rank Fusion with Multiquery Optimization for Streaming Search\"\n- **Medical RAG:** \"MedCPT\" (medical contrastive pre-training)\n\n---\n\n**Last Updated:** 2026-05-31\n
