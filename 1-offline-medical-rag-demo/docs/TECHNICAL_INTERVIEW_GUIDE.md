## 🎯 10 Core Interview Questions

### **Q1: Explain your project in 2 minutes**

**Strong Answer:**
> "I built an offline RAG (Retrieval-Augmented Generation) system for medical Q&A. RAG means I retrieve relevant documents first, then use them as context for answer generation—this reduces hallucinations compared to vanilla LLMs. The system is fully local: documents get embedded using sentence-transformers, stored in FAISS (a vector database), and answers are generated by flan-t5-small without any cloud APIs. The key insight is that grounding answers in actual documents makes them more reliable than an LLM just making things up."

**Why it works:**
- ✅ Clear definition of RAG
- ✅ Explains the problem it solves (hallucinations)
- ✅ Mentions tech stack concisely
- ✅ Highlights the key benefit (grounding)

**Strong Answer:**

It's lightweight (~50 MB), open-source, and widely used in production systems at Meta and Google. For this demo with ~1K documents, FAISS is plenty. If this scales to millions of documents, I'd consider managed solutions, but starting local lets me own the entire pipeline and understand how similarity search works at the index level."

### **Q3: Walk me through the data flow when a user asks a question**

**Strong Answer:**
> "When a user asks 'What is type 2 diabetes?', here's what happens:
> 
> 1. **Embedding:** The question gets embedded using sentence-transformers into a 384-dimensional vector.
> 2. **Retrieval:** That vector is passed to FAISS, which does fast similarity search and returns the top-3 most relevant chunks.
> 3. **Context Formatting:** Those chunks are combined into a context string (max 2000 chars) with source labels.
> 4. **Prompt Engineering:** I build a prompt: 'Answer ONLY using the provided context. [context]. Question: [question]'.
> 5. **Generation:** flan-t5-small generates an answer with a 128-token limit to stay concise.
> 6. **Return:** The answer is returned with source information (document name, chunk ID, similarity score).
>
> The critical piece: sentence-transformers and the embedding model must be the same for retrieval and query—if I used different embeddings, similarity search breaks."

**Why it works:**
- ✅ Step-by-step walkthrough (easy to follow)
- ✅ Mentions specific numbers (384-dim, 2000 chars, 128 tokens)
- ✅ Includes prompt engineering (shows sophistication)
- ✅ Highlights a subtle but crucial point (consistent embeddings)

### **Q4: Why normalize embeddings before inserting into FAISS?**

**Strong Answer:**
> "Good question. By default, sentence-transformers outputs unnormalized embeddings. I normalize them to unit length (L2 norm) before inserting into FAISS's IndexFlatIP (inner product) index. Here's why: with normalized vectors, inner product equals cosine similarity—which is what we want for semantic similarity. Without normalization, IndexFlatIP would compute raw dot products, which are sensitive to vector magnitude and don't reflect true semantic similarity. So normalization is a one-line preprocessing step that makes similarity search semantically correct."

**Why it works:**
- ✅ Shows understanding of linear algebra (L2 norm, dot product)
- ✅ Explains the relationship between normalization and similarity metrics
- ✅ Mentions the specific FAISS index type (IndexFlatIP)
- ✅ Concise but technical

**Interviewer might probe:**
- "What if you didn't normalize?" → Answer: "You'd get incorrect rankings; vectors with larger magnitudes would artificially score higher."
- "Could you use a different FAISS index?" → Answer: "Yes, like IndexFlatL2 for Euclidean distance, but inner product on normalized vectors is faster and more standard for embeddings."

### **Q5: How do you handle chunking? Why overlap?**

**Strong Answer:**
> "Documents get split into 400-character chunks with 80-character overlap. Here's the logic:
>
> - **400 chars:** Roughly 100–150 tokens (sentence-transformers works well with this length).
> - **80-char overlap (20%):** Ensures context at chunk boundaries isn't lost. For example, if a sentence spans the boundary, the overlap captures both parts.
>
> Without overlap, the LLM might answer a question by retrieving an incomplete chunk. With overlap, semantic search captures complete ideas. I validate this with a simple check: `assert overlap < chunk_size`, otherwise overlap doesn't make sense.
>
> The downside is storage overhead (~20% larger index), but for ~1K documents, it's negligible. If I had 10M documents, I might reconsider."

**Why it works:**
- ✅ Explains both parameters (size and overlap)
- ✅ Gives reasoning (token counts, context preservation)
- ✅ Mentions validation logic
- ✅ Acknowledges trade-offs (storage vs. quality)

### **Q6: Describe the architecture pattern you used and why**

**Strong Answer:**
> "I used several patterns:
>
> 1. **Factory Pattern** (for index creation): `build_vector_store()` function creates and returns a FAISS index—decouples index construction from pipeline.
> 2. **Strategy Pattern** (for vector stores): The `retriever.py` accepts any vector store with a `search()` method—I could swap FAISS for ChromaDB without changing the retriever.
> 3. **Template Method Pattern** (for the RAG pipeline): `pipeline.py` defines the fixed sequence: Load → Chunk → Embed → Index → Retrieve → Generate.
> 4. **Lazy Initialization** (for models): Models only load when first needed (first query or index build), not at startup. This keeps CLI startup fast.
>
> Why these patterns? They follow SOLID principles:
> - **Single Responsibility:** Each module does one thing (document_loader, chunker, embedder, etc.).
> - **Open/Closed:** New retrieval strategies can be added without modifying existing code.
> - **Dependency Inversion:** High-level code depends on abstractions (vector_store interface), not concrete implementations."

**Why it works:**
- ✅ Names specific patterns and explains their purpose
- ✅ Shows how they connect to SOLID principles
- ✅ Demonstrates design thinking, not just coding

### **Q7: What's the biggest limitation of this system, and how would you fix it?**

**Strong Answer:**
> "The biggest limitation is **limited grounding ability**. flan-t5-small is small (~60M parameters) and sometimes:
> 
> - **Ignores instructions:** It might use outside knowledge even though I prompt 'Use ONLY context.'
> - **Refuses to answer:** If no relevant chunks are found, it might output generic text instead of 'I don't know.'
> - **Factual errors:** Small models hallucinate despite context—larger models are more reliable.
>
> **How to fix:**
> 1. **Upgrade the LLM:** Use flan-t5-large or a finetuned model—better instruction-following.
> 2. **Add retrieval reranking:** Use a cross-encoder to rerank top-10 chunks before generation—ensures the best chunks reach the LLM.
> 3. **Confidence scoring:** Return uncertainty estimates (e.g., 'I'm 60% confident in this answer').
> 4. **Human feedback loops:** Collect which answers users rate as helpful, finetune on those examples.
>
> I could implement reranking in 2–3 days; upgrading the LLM takes 1 day (just change the model name). I chose flan-t5-small for this demo to keep it lightweight, but production would use a larger model or an open LLM like Mistral."

**Why it works:**
- ✅ Honest about real limitations (shows maturity)
- ✅ Prioritizes the most impactful issue
- ✅ Provides concrete solutions with effort estimates
- ✅ Explains trade-offs (small model = faster, but less capable)

### **Q8: How do you ensure answers are grounded in the documents?**

**Strong Answer:**
> "I use three techniques:
>
> 1. **Prompt Engineering:** The prompt explicitly instructs: 'Answer ONLY using the provided context. If the context doesn't contain the answer, say you don't know.' This sets expectations.
>
> 2. **Context Formatting:** I pass chunks with metadata (doc_id, chunk_id) so the LLM 'sees' where information comes from. This encourages grounding.
>
> 3. **Source Tracking:** I return the similarity scores and source references alongside the answer. Users can verify if the answer is actually in the documents.
>
> **Limitations:**
> - flan-t5-small still sometimes ignores instructions and uses outside knowledge.
> - If no relevant chunks exist, the LLM fills the gap with generated text.
>
> **How I measure success:**
> - Manually test with questions that have clear answers in the corpus.
> - For out-of-domain questions ('What's 2+2?'), check if it says 'I don't know' or generates an answer.
> - A proper evaluation would involve human annotators rating factuality on a held-out test set."

**Why it works:**
- ✅ Explains concrete techniques (prompt, formatting, tracking)
- ✅ Acknowledges limitations
- ✅ Shows how to measure success (testing strategy)

### **Q9: How would you scale this to 1M documents?**

**Strong Answer:**
> "Scaling has three dimensions:
>
> **1. Index Size (1K → 1M documents):**
> - Current: FAISS IndexFlatIP (O(n) search, ~10ms per query).
> - Problem: 1M vectors × 384 dims × 4 bytes = 1.5 GB; search becomes slow.
> - Solution: Use FAISS IndexIVF (Inverted File Index)—clusters vectors, searches only nearest clusters. ~100x faster, slight accuracy loss.
>
> **2. Retrieval Quality (1M docs, more noise):**
> - Problem: Top-3 chunks might be less relevant in a noisy corpus.
> - Solution: Add reranking—retrieve top-100, use cross-encoder to rerank to top-3. This is slower but improves quality.
>
> **3. Deployment (from single Python process):**
> - Current: Single CLI, Streamlit UI running locally.
> - Solution: Move to API (FastAPI), containerize (Docker), scale with Kubernetes or managed services (Azure Container Apps).
> - Model serving: Use Ollama or vLLM for efficient batched inference.
>
> **Implementation order:**
> - Day 1: Switch to IndexIVF (quick win).
> - Day 2–3: Add reranking.
> - Days 4–5: FastAPI + Docker.
> - Days 6–10: Kubernetes or managed cloud.
>
> Total effort: ~2 weeks with a team of 2."

**Why it works:**
- ✅ Identifies real bottlenecks (index size, search speed, quality)
- ✅ Proposes specific solutions with trade-offs
- ✅ Provides realistic effort estimates
- ✅ Shows systems thinking (beyond just the code)

### **Q10: What would you do differently if you built this again?**

**Strong Answer:**
> "Three things:
>
> 1. **Use Pydantic for validation:** I use dataclasses, which are fine, but Pydantic would add automatic validation (e.g., ensure chunk_id format is correct, embedding dimension matches). This catches bugs earlier.
>
> 2. **Add proper logging:** Currently, I print to stdout. For production, I'd use Python's logging module with structured logs (JSON format) to track errors and performance. This makes debugging easier.
>
> 3. **Build from the start with caching:** The vector index and embeddings are expensive to recompute. I save them, but I'd design this more explicitly—use a versioned cache directory and validate checksums. If I change chunk size or embedding model, I'd invalidate the cache automatically.
>
> 4. **More comprehensive testing:** I have example tests, but I'd add property-based testing (using Hypothesis) to generate edge cases automatically. For example: 'For any document, the sum of chunk lengths minus overlaps equals the original length.' Testing like that catches subtle bugs.
>
> **Reasoning:** These are all 'pay-as-you-go' investments. Early-stage projects don't need them, but once you hit 10K users or 1M documents, they matter. I'd implement these incrementally as the project grows."

**Why it works:**
- ✅ Concrete improvements, not vague
- ✅ Shows knowledge of tools (Pydantic, logging, Hypothesis)
- ✅ Explains when each matters (scaling reasoning)
- ✅ Demonstrates maturity (iterative improvement mindset)

## 🎯 5 Follow-Up Questions an Interviewer Might Ask

### **Follow-Up 1: "How do you handle out-of-vocabulary words or misspellings in queries?"**

**Strong Answer:**
> "Great question. sentence-transformers is trained on general English, so it handles most common words well. For misspellings:
>
> - **Short queries:** sentence-transformers is robust to minor typos (e.g., 'diabetus' → 'diabetes'). The transformer can infer intent.
> - **Long queries:** Longer context helps; one misspelled word is diluted by correct words.
> - **Proper fix:** I could add a spell-checker preprocessor (e.g., TextBlob, Hunspell) before embedding, but that adds latency.
>
> For domain-specific terms (medical jargon), I'd:
> 1. Add a domain vocabulary file
> 2. Finetune sentence-transformers on medical docs (using a specialized corpus like PubMed)
> 3. Or: Switch to a medical BERT variant (e.g., SciBERT or BioBERT)
>
> Trade-off: Finetuning improves medical accuracy but requires labeled data (~10K examples) and 2–3 GPU hours. Not worth it for a portfolio project unless I have that data."

---

### **Follow-Up 2: "What happens if the user asks a question that requires reasoning across multiple documents?"**

**Strong Answer:**
> "Example: 'Can someone with both diabetes and hypertension take metformin?'
>
> Current system:
> - Retrieves chunks about diabetes AND chunks about hypertension separately.
> - LLM sees both in context and generates an answer.
> - This works for simple cases (both topics are in the top-3 results).
>
> **Limitations:**
> - If the answer requires connecting info from 4+ chunks, I might miss it.
> - No explicit reasoning; the LLM just does its best.
>
> **Fixes:**
> 1. **Multi-hop retrieval:** Retrieve top-5, use LLM to rerank by relevance to the original query. Iteratively retrieve chains of related chunks.
> 2. **Graph-based retrieval:** Build a knowledge graph where documents are nodes and semantically related chunks are edges. Follow edges to find multi-hop paths.
> 3. **Supervised finetuning:** Train on multi-hop questions and collect which chunks the LLM should have used.
>
> For this project, I'd try #1 (multi-hop retrieval) first—it's a 2-day implementation and requires minimal changes to the pipeline."

---

### **Follow-Up 3: "How would you evaluate the quality of answers? What metrics would you use?"**

**Strong Answer:**
> "This is hard because medical Q&A is subjective. Here are evaluation strategies:
>
> **1. Automated metrics (easy, biased):**
> - **BLEU/ROUGE:** Measure lexical overlap with reference answers. Fast but penalizes paraphrasing.
> - **BERTScore:** Semantic similarity using embeddings. Better than BLEU but still imperfect.
> - **Retrieval metrics:** Precision@K (are top-K chunks relevant?), MRR (mean reciprocal rank).
>
> **2. Human evaluation (slow, gold standard):**
> - Have 3+ medical professionals rate answers as: Correct / Partially Correct / Incorrect.
> - Measure inter-rater agreement (Cohen's Kappa ≥ 0.8 is good).
> - For 100 test questions, ~300 ratings (3 raters × 100 questions).
>
> **3. What I'd do for this project:**
> - Build a test set of 20–30 medical questions with reference answers.
> - Manually rate generated answers (I'd play the role of evaluator).
> - Track: Answer quality (subjective), source references (objective), hallucinations (objective).
> - Target: 80%+ of answers are correct, with sources, no hallucinations.
>
> **Proper approach (for production):**
> - Set up a feedback loop: users rate answers (👍 / 👎), collect data.
> - Use this data to finetune a reranker or the LLM.
> - Track metrics over time (do they improve after finetuning?).
> - A/B test different model sizes or retrieval strategies."

---

### **Follow-Up 4: "Could you use a simpler approach without embeddings? Like keyword search with TF-IDF?"**

**Strong Answer:**
> "Yes, absolutely. TF-IDF is simpler and would work for this domain. Let me compare:
>
> | Aspect | TF-IDF | Embeddings |
> |--------|--------|-----------|
> | Semantic understanding | No ('diabetes' ≠ 'blood sugar') | Yes (captures meaning) |
> | Misspelling tolerance | Low | Medium-High |
> | Training required | No (unsupervised) | No (pretrained models) |
> | Latency | ~1ms per query | ~50ms (inference) |
> | Memory | ~50 MB | ~500 MB (model + index) |
> | Accuracy | ~60% on medical Q&A | ~85% |
> | Scalability | O(n) with TF-IDF tricks | O(log n) with IVFADC |
>
> **When to use TF-IDF:**
> - You have small data, simple vocabulary.
> - Latency is critical (embed searches in mobile apps).
> - You want no dependencies (just numpy).
>
> **When to use embeddings (my choice):**
> - You have semantic variation (medical terms, paraphrasing).
> - Accuracy matters more than latency.
> - Infrastructure supports model serving.
>
> For a medical Q&A system, embeddings are better because medical language is complex (synonyms like 'diabetes' and 'DM'). TF-IDF would miss that. But for a quick MVP, TF-IDF + BM25 (a better version) is solid."

### **Follow-Up 5: "How do you handle privacy if someone uploads sensitive medical data?"**

**Strong Answer:**
> "This is critical. Current system:
> - Data stays local (no cloud upload).
> - Embeddings aren't reversible (can't extract text from a 384-dim vector).
> - But: You can extract chunks from the pickled vector store file.
>
> **Privacy measures I should add:**
> 1. **Encrypt the vector index:** Use AES-256 encryption on `chunks.pkl` and `index.faiss`.
> 2. **Access control:** Require authentication to query the index.
> 3. **Audit logging:** Log all queries and who made them (HIPAA compliance).
> 4. **Data minimization:** Don't store raw documents after indexing—only store what's needed.
> 5. **Retention policy:** Auto-delete indexes older than 30 days.
>
> **For this portfolio project:**
> - I'm using public medical documents (educational content), so privacy isn't critical.
> - In production (hospital data), I'd implement all 5 measures.
> - Also need: Role-based access, IP whitelisting, network isolation.
>
> **HIPAA compliance** (if handling patient data):
> - Requires encryption, audit logs, data deletion on request.
> - Managed services (Azure AI Search, AWS Kendra) handle this; building from scratch is risky.
> - I'd recommend using a HIPAA-compliant cloud service for sensitive data, not a local system."

## 🌟 Strong Explanations (Detailed)

### **Why This Architecture Was Chosen**

**The RAG Approach:**
> "I chose RAG over a vanilla LLM for three reasons:
>
> 1. **Hallucination reduction:** LLMs sometimes make up facts. By grounding answers in documents, I can guarantee accuracy. If the document says 'Metformin is used for type 2 diabetes', the answer comes from that sentence, not from the LLM's learned knowledge.
>
> 2. **Transparency:** With vanilla LLMs, users don't know where answers come from. RAG lets me return source references (document name, chunk ID, similarity score). Users can verify the answer themselves.
>
> 3. **Offline independence:** I don't depend on expensive LLM APIs (OpenAI, Anthropic). I can host locally and iterate without per-query costs. This is crucial for a portfolio—shows I can build the full pipeline.
>
> The trade-off: RAG is harder than just calling an API. But I wanted to demonstrate systems thinking and ownership of the entire stack."

**The Pipeline Design:**
> "I split the pipeline into independent modules (loader, chunker, embedder, vector_store, retriever, generator) for three reasons:
>
> 1. **Debuggability:** Each module does one thing. If answers are bad, I can isolate the issue: Is it wrong documents being retrieved? Or bad generation? I test each module separately.
>
> 2. **Swappability:** Want to try a different embedding model? Just change the `embedder.py` import. Want a different vector store? Drop-in replacement for `vector_store.py`. This flexibility is valuable for experimentation.
>
> 3. **Testability:** I can mock each module independently and write unit tests without running the entire pipeline. This catches regressions early.
>
> Alternative: Monolithic pipeline (everything in one file). Simpler to start, but unmaintainable as it grows."

**Why Lazy Initialization:**
> "Models are large (~100 MB) and startup is slow. Instead of loading at import time, I load on first use. This means:
> - `python main.py --help` runs in <50ms (instant feedback).
> - First query takes 2s (model loads), subsequent queries take 50ms.
> - This is important for CLI UX—users don't feel stuck on startup."

### **Project Limitations (Honest Assessment)**

**Limitation 1: Small LLM Hallucinations**
> "flan-t5-small is prone to hallucination. Even with the prompt 'Use ONLY provided context', it sometimes generates facts not in the documents. This is a hard problem—even GPT-4 does this. My workaround: explicit source tracking and human review for high-stakes queries. A proper fix requires a larger model, finetuning on medical Q&A, or guardrails (e.g., refusing if confidence is low)."

**Limitation 2: Limited to Training Data**
> "The system can only answer questions about documents it has seen. If a user asks about a drug not in the corpus, it hallucinates or says 'I don't know.' This is by design (offline), but it means the corpus must be curated. For a full medical encyclopedia, I'd need 10K+ documents and would hit scalability limits."

**Limitation 3: No Real-Time Updates**
> "Adding a new document requires rebuilding the FAISS index (takes ~10s for 1K docs). In production, I'd implement incremental indexing so new documents appear immediately. Current design doesn't support this."

**Limitation 4: No Personalization**
> "Every user gets the same answer to the same question. No user history, preferences, or context. For a medical app, personalization could be important (e.g., 'Give me answers in simple language' vs. 'Give me clinical details'). Current system treats all users the same."

**Limitation 5: Single-Language**
> "System is English-only. Medical knowledge is global; multilingual support would improve accessibility. But this requires multilingual embeddings and multilingual LLMs, adding complexity."

**Limitation 6: Cold-Start Problem**
> "First query takes 2s (model loads). Users on slow connections experience bad UX. For web apps, I'd preload models or use model caching servers (e.g., Ollama, vLLM)."

### **What Should Be Improved Next (Prioritized)**

**Priority 1 (Highest Impact, Shortest Time):**

1. **Add Confidence Scoring** (1 day)
   - Return a confidence score with each answer (0–100%).
   - If confidence < 50%, surface a disclaimer: 'This answer may be unreliable.'
   - Implement: Cross-encoder reranking or LLM self-confidence estimation.

2. **Implement Retrieval Reranking** (2–3 days)
   - Retrieve top-100 chunks, rerank to top-3 using a cross-encoder.
   - This improves quality by 10–15% without changing the LLM.
   - Tool: sentence-transformers cross-encoders or a specialized reranker.

**Priority 2 (Good Impact, Medium Time):**

3. **Upgrade LLM** (1 day)
   - Switch from flan-t5-small to flan-t5-large or Mistral 7B (if GPU available).
   - Reduces hallucinations, better instruction-following.
   - Trade-off: Slower (needs GPU).

4. **Add Multi-Hop Retrieval** (3–5 days)
   - Support questions requiring context from multiple documents.
   - Iterate: Retrieve chunk 1, use LLM to generate sub-queries, retrieve chunk 2, synthesize answer.
   - This handles complex questions better.

**Priority 3 (Nice to Have, Longer Time):**

5. **Comprehensive Evaluation Framework** (1 week)
   - Build test set of 50–100 questions with human-rated reference answers.
   - Track metrics: accuracy, hallucination rate, source attribution.
   - A/B test different models/strategies.

6. **FastAPI + Deployment** (1 week)
   - Move from CLI to a REST API.
   - Docker containerization.
   - Deploy on Cloud (Azure, AWS, GCP).

7. **Caching and Optimization** (3–5 days)
   - Implement query result caching (10K most common queries).
   - Model quantization (reduce size 4x, 10% accuracy loss).
   - Batch inference for multiple queries.

**Why This Order:**
- Priority 1 & 2 improve accuracy with minimal effort.
- Priority 3 improves deployment and robustness (needed for production).
- As a portfolio project, stopping after Priority 2 is strong; Priority 3 shows even more ambition.
